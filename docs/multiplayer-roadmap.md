# 多人連線狀態與路線

## 目前已完成

- 建立新局時可選擇 1 到 6 名真人玩家，其餘座位由 AI 補滿。
- 開局者會自動佔用第一個真人座位，前端保存該座位的 session token。
- 分享連結只包含 `game_id`，知道網址的人可以進入 lobby 並認領尚未被佔用的真人座位。
- 每個真人座位都有獨立 session token；私人視角與行動 API 都需要 `X-Player-Token`。
- Lobby 只顯示公開安全資訊：座位、名字、是否 AI、是否已認領、存活狀態。
- 正常遊戲 API 不會把其他玩家真實角色、陣營、私人情報或其他人的 session token 傳給前端。
- 投票改為多人流程：每位可投票真人分別投票或棄票，全部真人處理完後才由 AI 補票並結算。
- 真人可公開發言、私聊任何其他玩家、提名、投票、Artist 提問、Klutz 選人。

## 使用方式

1. 開局者在 setup 畫面選擇真人人數。
2. 建立新遊戲後，複製畫面上的分享連結。
3. 其他玩家用自己的瀏覽器或手機打開分享連結。
4. 每個玩家在 lobby 選一個開放真人座位並輸入名字。
5. 進入遊戲後，每個玩家只會看到自己的角色、私人情報、私聊與合法行動。

## 安全模型

- `TruthState` 只存在後端 domain layer。
- 正常玩家 API 每次都由 `ContextBuilder`/view builder 產生玩家視角，不直接序列化 ORM 或完整 state。
- Session token 只代表一個真人座位，不代表全局管理員權限。
- 如果某人拿到別人的瀏覽器 localStorage token，就等同拿到該座位權限；正式公開服務前需要 HTTPS 與基本房間保護。
- Postgame reveal 仍然只在 `GAME_OVER` 後開放。

## 仍待加強

- 即時推送：目前是輪詢與手動 AI tick，下一步應加入 SSE 或 WebSocket，只推送該 session 可見的事件。
- 房間保護：加入房間邀請碼、旁觀者模式、主持人踢人/鎖座位。
- 斷線恢復：目前依靠同一瀏覽器 localStorage；跨裝置恢復需要玩家輸入短碼或登入。
- 並發鎖：目前投票與一般 API 由後端驗證，未來公開部署時應加入 per-game action lock，避免高延遲下雙重操作。
- 更多真人人數與劇本：目前仍以六人 No Greater Joy 為核心，之後可擴展到更多劇本與 7 人以上配置。

## 建議下一步

1. 加入房間邀請碼與觀戰者權限。
2. 加入 SSE/WebSocket，將公開事件與私人事件分流推送。
3. 把夜間行動做成更完整的多真人等待面板。
4. 加入 mobile-first 的座位與聊天版面。
5. 擴充 Trouble Brewing 或自訂劇本資料模型。
